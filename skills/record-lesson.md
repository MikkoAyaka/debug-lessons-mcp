# Record Lesson / 记录踩坑

将调试经验写入共享案例库。三种模式自动适配：

| 调用方式 | 模式 | 行为 |
|----------|------|------|
| `/record-lesson` | **自主总结** | 回顾当前对话，提取所有踩坑经历，全量入库 |
| `/record-lesson 问题描述` | **快速录入** | 从描述中提取信息，一次提交 |
| `/record-lesson docs/pitfalls.md` | **批量导入** | 读取文件，解析所有条目，连续导入不打断 |

---

## 模式 A：自主总结（无参数）

当用户只输入 `/record-lesson` 不带任何参数时：

1. **回顾对话**：扫描当前会话中所有调试经历、错误修复、踩坑发现
2. **提取案例**：每个独立问题作为一个案例，从对话中还原 problem → thinking → result → lesson
3. **批量入库**：对每条连续调用 `add_case`，不暂停询问，全部导完
4. **简洁汇报**：

```
✅ 从本次对话中总结了 N 条踩坑案例：
  [ID] 分类 — 问题摘要
  [ID] 分类 — 问题摘要
  ...
```

### 识别踩坑的信号

对话中出现以下特征时应被提取：
- 用户报告了错误/异常/失败
- 经历了多次尝试才解决问题
- 用户表达了意外/困惑（"为什么..."、"奇怪..."、"居然..."）
- 最终找到了具体原因和解决方案
- 用户总结了经验教训（"原来..."、"以后..."、"记住..."）

---

## 模式 B：快速录入（参数是问题描述）

1. 从用户输入中提取关键信息，不逐项询问
2. 信息完整就直接调用 `add_case`，**不确认**
3. 只有明显缺失时才补问（最多一个问题）

---

## 模式 C：批量导入（参数是文件路径）

当参数指向现有文件（`.md`、`.json`、`.txt`）：

1. 读取整个文件
2. 解析所有案例条目
3. 连续调用 `add_case`，**不逐条确认，不问"继续吗"**
4. 汇报汇总

---

## 通用规则

- 案例用中文记录
- 教训具体可操作，不说「下次注意」
- `model_id` 无需传入——服务端自动检测
- `project_id` 不传（服务端自动检测）
- `category` 从问题领域推断

---

## 环境触发锚点

在正常对话中（即使用户未显式调用本命令），出现以下信号时主动建议记录：
- 用户说 "终于搞定了" / "原来是这个原因" / "记住这个坑"
- 一个 Bug 被修复且原因非显然
- 经历了 ≥3 次尝试才解决的问题
